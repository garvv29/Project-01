Daily Design Journal · Day 4
Section 1 – Specific Bug

While adding a PRINT command to Redis Lite, I had to implement a corresponding print() function inside my HashMap. That's where I started getting compiler errors.

error: binding 'const SinglyList<...>' to reference of type 'SinglyList<...>&' discards qualifiers

After fixing that, another error appeared.

terminate called after throwing an instance of 'std::out_of_range'
what(): SinglyList: INVALID INDEX

Then while trying another fix I got:

error: 'Entry' has no member named 'size'

The actual problem was in the way I was iterating through the buckets inside the print function.

Section 2 – Failed Attempt

At first I thought the print function itself was correct and the exception was coming from the linked list again. So I spent some time checking the linked list implementation before realising the iteration logic inside HashMap was wrong.

I was treating an Entry object as if it was another linked list and even tried calling size() on it. After reading through the code carefully I noticed I had mixed up the bucket and the entry while writing the loops.

Later while reviewing everything I also noticed another limitation. My print() function only worked properly for built-in data types. I first tried thinking of ways to pass a print function into HashMap automatically, but that would make every use of the class more complicated. After looking into it, overloading operator<< for custom types looked like the cleanest solution.

Section 5 – Learning Reflection

Today wasn't about adding a lot of new features. Most of the time went into cleaning up the project and finding small bugs that only showed up after everything started working together.

One thing I realised is that utility functions like print() are easy to underestimate. The HashMap itself was working fine for inserts and lookups, but printing every element meant traversing every bucket and every linked list correctly. A small mistake in those loops was enough to throw exceptions even though the stored data itself was valid.

I also understood why generic libraries can't magically print user-defined classes. For built-in types it's easy because << already exists, but for custom classes the compiler has no idea how they should be displayed. That made me understand why STL streams rely on overloading operator<<.

Apart from fixing bugs, I also spent some time reading how Redis actually works internally. I started another small Redis project separately just for learning, where I was looking into sockets, client-server communication, modular project structure and basic multithreading in C++. It isn't part of this submission yet, but it helped me understand how the data structures I'm building could actually be used in a real application.

By the end of the day the Redis Lite implementation was mostly complete, the PRINT command was working after fixing traversal bugs, I had reviewed the existing codebase for small issues, and I had also finished writing the remaining design journals for the previous development days.
