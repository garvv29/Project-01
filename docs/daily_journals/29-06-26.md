Daily Design Journal · Day 5
Section 1 – Specific Bug

Most of today's work was on networking and testing.

The first problem was that my socket server wasn't accepting connections from another laptop even though everything worked locally using 127.0.0.1. After checking different things I found that Windows Defender was blocking the connection.

Another issue came while testing communication between the client and server. Longer messages were sometimes getting mixed together or not being parsed correctly by the server because I wasn't marking where one message ended.

Later while setting up GoogleTest I also ran into multiple build errors because my compiler and GTest versions weren't compatible.

Section 2 – Failed Attempt

Initially I thought there was something wrong with my socket code because another laptop couldn't connect to the server. I spent some time checking ports and IP addresses before realising Windows Defender was blocking incoming connections. After changing the firewall settings, the connection started working.

For parsing messages, I first assumed recv() would always give me one complete message. After testing a bit more I realised TCP doesn't work like that. Sometimes multiple messages arrive together or a message gets split. I started thinking about using a delimiter like \n so the server knows exactly where one command ends.

The GoogleTest setup also took much longer than I expected. At first I kept reinstalling different things because I thought I had linked the libraries incorrectly. Later I found that my MinGW installation and C++ version were outdated, so I switched to MSYS2, updated the compiler and installed a newer version of GoogleTest through it. After updating everything and rebuilding with CMake, it finally started working.

Section 5 – Learning Reflection

Today was quite different from the previous days because I wasn't really working on data structures anymore. Most of the time went into understanding networking and setting up a proper testing environment.

The biggest thing I learnt was that just because a socket program works on the same machine doesn't mean it'll work over a network. There are other things involved like firewall rules, IP addresses and how the operating system handles incoming connections. That was something I hadn't really thought about before.

I also understood that TCP is a stream, not a message-based protocol. I was expecting every send() to match one recv(), but that's not guaranteed. That made me realise why protocols usually define delimiters or packet formats instead of assuming every message arrives separately.

Setting up GoogleTest also taught me that build tools matter just as much as code. I spent more time fixing my development environment than actually writing tests, but once everything was configured correctly it became much easier to verify my data structures instead of relying only on manual testing.

By the end of the day I had a basic client-server setup working with Winsock, my Redis server could receive and parse commands from a client, GoogleTest was successfully integrated into the project using CMake, and I had written and run test cases for the Dynamic Array, Singly Linked List implementations.
